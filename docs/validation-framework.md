# 调研与验证框架

## 一、结论标签

后续所有结论统一打标签：

- `已验证`：本人已实际跑通
- `半验证`：工具已试通，但关键链路未完整验证
- `待验证`：只有资料和推断，没有实际操作
- `不建议`：风险或成本明显不划算

## 二、单项调研流程

1. 明确客户场景
2. 选定候选技术栈
3. 跑最小 demo
4. 记录配置与阻塞点
5. 输出成本、风险、可复用性结论

## 三、统一输出模板

每个能力至少回答下面 6 个问题：

| 问题 | 说明 |
| --- | --- |
| 解决什么问题 | 直接对应客户使用场景 |
| 用什么工具拼出来 | 开源、SaaS、自建分别是什么 |
| 最小 demo 怎么搭 | 最短路径，不追求完美 |
| 交付给客户长什么样 | 截图、视频、账号、后台、报告 |
| 主要风险是什么 | 技术、合规、风控、成本 |
| 后续怎么平台化 | 哪些部分能沉淀为共用能力 |

## 四、建议统一字段

后续做表格或数据库时，建议统一以下字段：

- 能力名称
- 优先级
- 目标客户
- 使用场景
- 候选工具
- demo 状态
- 月底交付物
- 单客户交付成本
- 平台化潜力
- 主要风险
- 下一步动作
