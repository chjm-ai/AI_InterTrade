# 项目总览

## 1. 项目目标

在有限时间内，完成外贸行业 4 类 AI 工具方向的可行性评估，并沉淀出可交付的 demo 材料。

本项目关注的是：
- 技术能不能跑通
- 工具能不能被客户真正用起来
- 是否值得继续封装成平台能力

本项目不关注的是：
- 完整产品研发
- 大规模工程化
- 复杂品牌包装

## 2. 统一判断标准

每个方向都按同一套标准评估：

| 维度 | 要回答的问题 |
| --- | --- |
| 客户价值 | 客户为什么愿意买单，而不是自己用现成工具 |
| 技术可行性 | 第三方工具和现有能力能否快速拼出 demo |
| 交付难度 | 是否能在前期平台不完善时完成手工交付 |
| 可复用性 | 后续能否沉淀为平台通用模块 |
| 风险 | 是否容易踩平台风控、稳定性或渠道接入问题 |

## 3. 四个能力的定位

### A. 多平台发布

定位：内容矩阵分发工具的轻封装版本。

价值假设：
- 客户不想理解复杂后台，只想批量发内容
- 可作为客户侧“看得见”的表层能力

核心问题：
- 和传统矩阵发布工具相比，差异化是什么
- 内容如何从自家系统流入
- 是否支持账号矩阵和多客户隔离

### B. 爆款内容拆解

定位：抓取行业内容后做结构化分析，辅助选题和模仿。

价值假设：
- 客户想要的是“给我结论”，不是原始爬虫数据
- 分析结果可以反哺内容生成和发布

核心问题：
- Apify 成本是否可控
- 抓取字段是否稳定
- 结果如何映射到后续平台或 skill 工作流

### C. 多平台多语种询盘承接

定位：把多个渠道的客户消息统一接入，再做 AI 辅助回复和分配。

价值假设：
- 这是最接近外贸成交链路的能力
- 客户对“询盘不漏接、能多语种承接”有明确价值感知

核心问题：
- Chatwoot 接哪些渠道最现实
- 多客户隔离如何处理
- Webhook 与后台配置能否足够轻量

### D. 评论截流

定位：抓取评论线索，用关键词或 AI 判断后再人工跟进。

价值假设：
- 适合做补充获客能力
- 可与内容分析形成闭环

核心问题：
- 各平台验证成本太高
- 风控风险较大
- 前期不一定适合做主打能力

## 4. 当前结论

### 应优先推进

多平台多语种询盘承接。

理由：
- 最接近成交和 ROI
- 更容易形成 demo 与视频
- 适合后续扩展为统一消息中心

### 应同步调研

爆款内容拆解。

理由：
- 适合做成标准化分析服务
- 可沉淀为 skill、报告、平台模块三种形态

### 应延后包装

多平台发布、评论截流。

理由：
- 前者容易显得“只是工具代装”
- 后者验证成本和风险都偏高
