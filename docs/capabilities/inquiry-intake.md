# 多平台多语种询盘承接

## 结论状态

`待验证（优先）`

## 1. 能力定义

把来自多个平台的客户询盘统一接入一个消息中心，自动识别语言、生成回复建议、支持人工接管，并沉淀客户线索与跟进记录。

## 2. demo 目标

本轮不是做完整产品，而是做一个客户看得懂、内部可复现的最小 demo。

### 本轮要证明

| 要证明的点 | 说明 |
| --- | --- |
| 多平台统一接入可行 | 至少 2 个渠道的消息能进入同一收件箱 |
| 多语种承接可行 | 英文、西语询盘可被识别并生成建议回复 |
| AI 辅助而非纯自动回复 | 支持客服审核后发送，降低误回复风险 |
| 线索分配可演示 | 新询盘可按规则进入指定团队或人员 |
| 能沉淀成平台模块 | 后续可拆成渠道接入、翻译、回复建议、CRM 同步四层 |

### 本轮不做

- 不追求全渠道覆盖
- 不做复杂权限系统
- 不做完整 CRM
- 不做高风险平台自动化抓取
- 不承诺无人值守全自动成交

## 3. 目标客户场景

| 场景 | 痛点 | demo 中如何体现 |
| --- | --- | --- |
| 外贸团队同时运营多个平台 | 消息分散，容易漏回 | 2 个渠道统一进入 Chatwoot |
| 客服语种能力有限 | 看不懂小语种，回复慢 | AI 自动识别语言并给出中文理解和目标语回复建议 |
| 询盘高峰时分配混乱 | 不知道谁跟进了哪条线索 | 用标签或收件箱规则分配 |
| 老板想看 ROI | 只想看到“有没有漏接、能不能更快回复” | 演示响应时间、客户资料、会话状态 |

## 4. 最小可执行 demo 范围

## 渠道选择

优先只做 2 个低阻力渠道：

| 渠道 | 角色 | 选择原因 | 状态标签 |
| --- | --- | --- | --- |
| WhatsApp Cloud API | 主渠道 | 外贸客户认知强，官方 API 相对标准 | `待验证` |
| 网站询盘表单 / 独立站聊天入口 | 辅渠道 | 最容易自控，适合先跑通 Webhook | `可优先验证` |

备选，不纳入第一轮主链路：

- Facebook / Instagram DM：价值高，但授权与接入步骤更繁琐
- 邮件：可做补充，但演示效果不如即时消息直观
- Alibaba / Made-in-China / RFQ 类平台：价值高，但平台权限和账号条件不确定，先不作为首轮 demo

## 核心链路

```text
客户从 WhatsApp 或网站表单发来询盘
-> Chatwoot 统一收件
-> Webhook 推送到 AI 中间层
-> AI 识别语言、提取意图、生成建议回复
-> 结果回写 Chatwoot 备注 / 私有 note / 标签
-> 客服审核后发送
-> 会话标记状态并进入线索跟进表
```

## 5. 建议技术栈

| 层 | 方案 | 用途 | 备注 |
| --- | --- | --- | --- |
| 消息中心 | Chatwoot | 统一收件箱、客服界面、标签、指派 | demo 主后台 |
| 渠道接入 A | WhatsApp Cloud API | 接入 WhatsApp 消息 | 若申请受阻，可先用网站渠道替代主演示 |
| 渠道接入 B | Web Form + Webhook | 接收网站询盘 | 最稳定的保底链路 |
| AI 中间层 | n8n 或轻量 Node/Python webhook 服务 | 编排语言识别、翻译、回复建议 | demo 期优先低代码 |
| 大模型 | OpenAI / OpenRouter 任一稳定模型 | 语言识别、翻译、回复建议、字段提取 | 先选调用最稳的 |
| 线索沉淀 | Google Sheets / Airtable | 存 demo 线索台账 | 够演示即可 |

## 7. 消息处理规则

## 输入

- 客户原始消息
- 渠道来源
- 联系方式
- 时间戳
- 会话上下文

## AI 输出

| 输出字段 | 用途 |
| --- | --- |
| `language` | 识别客户语种 |
| `intent` | 判断是报价、打样、MOQ、交期还是售后 |
| `summary_cn` | 生成中文摘要给销售看 |
| `reply_draft` | 生成对应语种的建议回复 |
| `priority` | 标记高优先级询盘 |
| `missing_info` | 提示客户还缺哪些关键信息 |

## 建议规则

1. 新消息进入后先打 `new-lead`
2. AI 判断为报价类则追加 `price-request`
3. AI 判断为打样类则追加 `sample-request`
4. 若消息包含明确采购数量、交期、国家信息，优先级升高
5. 所有 AI 内容先写入内部 note，由人工确认后再回复

## 9. 演示脚本建议

建议按下面顺序录制：

1. 先展示两个渠道入口
2. 从网站表单提交一条英文询盘
3. 从 WhatsApp 发一条西语询盘
4. 展示消息进入同一收件箱
5. 打开会话，展示 AI 生成的中文摘要、标签、回复建议
6. 人工修改一句回复并发送
7. 展示线索同步到台账
8. 结尾说明可扩展到更多平台和更多语种

## 10. 成功标准

| 指标 | 通过标准 |
| --- | --- |
| 渠道接入 | 至少 2 个来源消息进入同一收件箱 |
| 多语种识别 | 英文、西语样本识别正确 |
| 回复辅助 | 至少 3 条样本能生成可用回复草稿 |
| 人工接管 | 客服可编辑后发送，不依赖全自动 |
| 可复现性 | 能按文档重新跑一次 |
| 可销售性 | 录屏能在 3 分钟内讲清价值 |

## 11. 风险与替代方案

| 风险 | 影响 | 替代方案 | 标签 |
| --- | --- | --- | --- |
| WhatsApp 审核或配置耗时 | 主链路延期 | 先用网站表单 + 邮件 / 模拟 webhook 演示 | `高` |
| Chatwoot 渠道适配受限 | 无法统一接入 | 用 Chatwoot 做主界面，渠道先通过自建 webhook 转入 | `中` |
| AI 回复不稳定 | 销售不敢用 | 强制人工审核，不启用自动发送 | `中` |
| 小语种样本不足 | 演示说服力不够 | 先聚焦英语 + 西语 | `低` |

## 12. 后续平台化拆分

后续如果验证通过，可以沉淀成 4 个模块：

| 模块 | 说明 |
| --- | --- |
| 渠道连接器 | WhatsApp、表单、邮件、社媒私信统一接入 |
| AI 询盘助手 | 语言识别、摘要、回复建议、字段提取 |
| 线索分配引擎 | 标签、优先级、销售归属、SLA |
| 客户配置中心 | 每个客户自己的渠道、提示词、签名、知识库 |
